# Архитектура и стек

## Цель архитектуры

Сделать минималистичное readonly-зеркало сайта “Последнее слово”, которое:

- просто поддерживать небольшой командой;
- можно полностью собрать статически;
- не требует сервера на production;
- не дублирует WordPress-админку;
- сохраняет консистентную связь между WordPress и репозиторием;
- позволяет постепенно добавлять поиск, статистику и инструменты анализа.

Главный принцип: сложность должна жить в импорте и сборке, а опубликованный сайт должен оставаться максимально простым.

## Общая схема

```text
WordPress
  ↓
import script
  ↓
local content files
  ↓
Astro build
  ↓
static HTML/CSS/JS
  ↓
GitHub Pages
```

WordPress остается источником истины. Репозиторий хранит подготовленную копию данных, которая нужна для статичной сборки и анализа.

Дата последней синхронизации хранится отдельно в `src/data/wpExportMeta.json`. Она используется
в шапке сайта для строки “База актуальна на ...” и на главной странице для расчета периода
“Последние добавления за неделю”.

## Production runtime

На GitHub Pages нет сервера, базы данных и записи на диск.

Поэтому production-сайт:

- только читает уже собранные данные;
- не сохраняет пользовательские изменения;
- не обращается к WordPress из браузера;
- не требует авторизации;
- не зависит от доступности WordPress после сборки.

Это делает сайт надежным: если WordPress временно недоступен, уже опубликованное зеркало продолжает работать.

## Стек

Основной стек:

- Astro - статичная сборка страниц;
- TypeScript - типизация данных, импортов и helper-функций;
- plain Markdown - хранение текстов последних слов;
- YAML frontmatter - метаданные записей;
- GitHub Pages - хостинг статичного сайта;
- GitHub Actions - автоматический импорт и сборка.

Опционально позже:

- SQLite - локальный промежуточный слой для сложных связей и отчетов;
- Pagefind/FlexSearch/MiniSearch - клиентский full-text поиск;
- отдельные Node.js-скрипты - импорт, нормализация, проверки качества данных.

## Хранение данных

Базовый и самый простой вариант: одно последнее слово = один Markdown-файл.

```text
src/content/last-words/
  12345-ru.md
  12345-en.md
  67890-ru.md
```

В frontmatter лежат структурные данные. В теле файла лежит текст последнего слова.

Такой формат удобен, потому что:

- его легко читать в diff;
- его можно редактировать локально без специальных инструментов;
- Astro Content Collections умеют валидировать схему;
- файлы хорошо ложатся в Git;
- текст не смешивается с кодом.

Минимальный набор стабильных полей:

```yaml
wordpressId: 12345
wpSlug: example-person-ru
localSlug: example-person-ru
translationGroupId: "12345"
person: "Имя Фамилия"
title: "Последнее слово Имени Фамилии"
personDescription: "Описание человека"
caseDescription: "Описание дела"
country: "Страна"
period: "2020-е"
language: "ru"
originalLanguage: "ru"
sourceUrl: "https://..."
publishedAt: "2026-05-11T14:22:50"
modifiedAt: "2026-05-11T14:27:42"
statementDate: "2026-04-22"
readingTime: "5 минут"
tags:
  - "свобода слова"
featuredImage:
  url: "https://..."
  alt: "Имя Фамилия"
  mimeType: "image/jpeg"
  width: 1000
  height: 800
  sizeBytes: 195313
```

`wordpressId` обязателен. Он нужен для синхронизации, повторного импорта и сверки с
WordPress.

`language` описывает язык записи на сайте-зеркале. В текущей рабочей базе это русскоязычные
записи. `originalLanguage` описывает язык, на котором последнее слово было произнесено или
написано изначально. Если русская запись содержит перевод и ниже в том же тексте приведен
оригинал последнего слова на другом языке, `originalLanguage` получает код языка оригинала.
Для записей, где русский текст и есть оригинал, используется `originalLanguage: "ru"`.
Признак перевода не хранится отдельным полем: он вычисляется как
`language !== originalLanguage`.

## Рабочие выборки

Пока модель переводов и языка оригинала не доведена до конца, основные публичные рабочие
страницы разделяют контент временными правилами.

Русскоязычная рабочая выборка:

- `language === "ru"`;
- `sourceUrl` не содержит `/en/`.

Это правило используется на главной странице для счетчиков людей и стран, в базе `/words/`, в
отчетах по именам и при проверке отсутствующей даты произнесения. Оно нужно из-за текущего
переходного состояния контента, где рядом с каноническими `*-en.md` есть временные `*-ru.md`,
ведущие на английские WordPress URL.

Англоязычная каноническая выборка для текущих счетчиков:

- `language === "en"`;
- `sourceUrl` содержит `/en/`.

Базовая статистика главной страницы считается в `src/lib/statistics.ts`, а не прямо в Astro
шаблоне. Сейчас она включает каноническое количество слов, разбивку по языкам, количество
уникальных русскоязычных людей и список стран русскоязычной рабочей выборки. Эти значения
закреплены snapshot-тестом `npm test` / `npm run test:stats`; ожидаемые цифры лежат в
`scripts/statistics-expectations.json` и должны обновляться только вместе с осознанным
изменением базы или правил фильтрации.

Целостность текстов проверяется отдельно через `npm run test:integrity`. Baseline хранится в
`scripts/content-integrity-baseline.json` и содержит для каждого Markdown-файла `wordpressId`,
язык, `sourceUrl`, SHA-256 очищенного текста, длину текста и количество слов. Этот baseline
покрывает все локальные Markdown-файлы, а не только канонические записи главной страницы.

Русскоязычные теги сейчас дополнительно фильтруются по наличию кириллицы. Это временная
эвристика. После введения явных полей для языка оригинала, переводов и, при необходимости,
языка тегов эти правила нужно заменить структурной фильтрацией.

## Почему не база данных в production

Для GitHub Pages база данных в runtime не подходит: сайт не может выполнять серверный код и писать на диск.

SQLite можно использовать как build-time инструмент:

- собрать данные из WordPress;
- нормализовать связи;
- проверить дубликаты;
- построить отчеты;
- затем сгенерировать Markdown, JSON или статичные страницы.

Но SQLite не должен быть обязательной частью опубликованного сайта. Иначе архитектура станет сложнее без реальной пользы для readonly-зеркала.

## Импорт из WordPress

Импорт разделен на два Node.js-скрипта.

Первый шаг сохраняет JSON из WordPress REST API на диск. Второй шаг читает уже сохраненный JSON
и генерирует Markdown-файлы. Это нужно, чтобы не дергать WordPress и Deflect при каждом тесте
импорта.

Ответственность fetch-скрипта:

- получить записи из WordPress REST API;
- забирать только измененные записи через `modified`;
- сохранить читаемый UTF-8 JSON.

Ответственность import-скрипта:

- прочитать локальный JSON;
- сохранить `wordpressId`;
- остановиться до записи файлов, если во входном JSON или существующем Markdown-контенте есть
  дубликаты `wordpressId`;
- дописать инкрементальный JSONL-журнал импорта, если передан `--log-output`: одна строка на
  каждое созданное, обновленное или удаленное слово;
- нормализовать HTML/текст;
- извлечь основные поля из ACF и таксономий WordPress;
- сохранить данные featured image: URL, alt, MIME-тип, ширину, высоту и вес в байтах;
- очистить WordPress HTML от встроенных блоков “Поделиться в соцсетях”;
- определить язык конкретной записи (`language`) по соотношению кириллицы и латиницы в тексте;
- извлечь дату произнесения последнего слова в `statementDate` из нижнего служебного блока
  рядом с местом произнесения и источниками; поддерживаются русские даты, английские даты,
  `YYYY-MM-DD` и числовые форматы вида `19.08.2015`;
- разложить записи по Markdown-файлам;
- не ломать ручные поля, которых нет в WordPress;
- сформировать отчет об изменениях.

После импорта обновляется `src/data/wpExportMeta.json`: там фиксируются источник, `fetchedAt`,
часовой пояс и путь к сырому JSON. Frontend не обращается к WordPress напрямую, а использует эту
метаинформацию только во время статичной сборки.

Журнал действий импорта хранится в `src/data/wpImportLog.jsonl`. Формат - JSON Lines: одна
строка на одно действие `create`, `update` или `delete`. У строк одного запуска общий `runId`,
а сервисная страница `/service/import-log/` читает этот файл во время сборки и показывает
историю импортов.

На главной странице блок “Последние добавления за неделю” считается не от даты самой свежей
записи, а от даты синхронизации: берутся 7 полных календарных дней перед днем синхронизации в
часовом поясе из `wpExportMeta.timeZone`.

Текущий скрипт:

```sh
npm run fetch:wp -- --limit 20 --output data/wp-export/posts.latest.raw.json
npm run import:wp -- --input data/wp-export/posts.latest.raw.json --limit 20
npm run fetch:wp -- --all --limit 100 --delay 1500 --output data/wp-export/posts.all.raw.json
npm run import:wp -- --input data/wp-export/posts.all.raw.json --all
npm run import:wp -- --input data/wp-samples/posts.latest-20.raw.json --limit 20
```

`import:wp` не должен ходить в сеть. Он всегда работает только с локальным JSON через `--input`.
Для полной выгрузки используется `fetch:wp --all`: скрипт идет по страницам WordPress REST API,
сохраняет опубликованные записи в один UTF-8 JSON и делает паузу между запросами через
`--delay`.
Имена content-файлов генерируются в формате `<wordpressId>-<language>.md`, например
`19397-ru.md`. Такой формат не конфликтует с внутренними id Astro Content Collections.

GitHub Actions использует два режима синхронизации:

- `incremental`: берет записи, измененные после `src/data/wpExportMeta.json.fetchedAt`, и
  обновляет соответствующие Markdown-файлы;
- `full`: забирает все опубликованные записи и запускает импорт с `--prune`, чтобы удалить
  локальные Markdown-файлы, которых больше нет среди опубликованных WordPress-постов.

JSON-ответы WordPress не стоит сохранять на диск ровно в том виде, в котором их вернул
`curl`: кириллица может быть записана как `\u043f\u0440...`. Это валидный JSON, но его
неудобно читать и ревьюить. Для sample-файлов и промежуточных артефактов нужно сначала
распарсить ответ, а потом записать его обратно в UTF-8:

```js
const data = await response.json();
await fs.writeFile(file, JSON.stringify(data, null, 2) + "\n", "utf8");
```

Важно разделить поля на два типа:

- поля из WordPress, которые можно перезаписывать при импорте;
- локальные поля, которые команда может вести вручную.

Например, связи между словами и уточненные группировки могут быть локальными, если их нельзя
надежно получить из WordPress.

## Консистентность данных

Для консистентности нужны три уровня защиты.

1. Стабильные идентификаторы

`wordpressId` связывает локальную запись с оригиналом. `localSlug` отвечает только за URL и
может меняться реже, но не должен быть единственным идентификатором.

2. Валидация схемы

Astro Content Collections должны проверять обязательные поля, типы дат, языки и массивы тегов во время сборки.

3. Проверки качества

Отдельные скрипты должны находить:

- записи без источника;
- битые связи;
- отсутствующие переводы;
- пустые или слишком короткие тексты;
- теги с разным написанием.

Критическая проверка `wordpressId` уже вынесена в `npm run validate:content` и автоматически
запускается перед `npm run build`. Сборка падает, если у Markdown-записи нет числового
`wordpressId` или если один `wordpressId` встречается в нескольких файлах. Временное
исключение сделано только для переходных пар одного английского WordPress URL: два файла с
одним `wordpressId`, разными `language` и одинаковым `sourceUrl`, содержащим `/en/`. Такие
пары появились до окончательной чистки английского потока и должны исчезнуть после full import
с `--prune`; любые другие дубли остаются ошибкой.

Snapshot-проверка статистики вынесена в `npm test`. Она читает Markdown-контент напрямую,
сверяет текущие счетчики с `scripts/statistics-expectations.json` и отдельно ловит случайные
локальные копии content-файлов вида `* 2.md`, которые уже приводили к удвоению английских
записей в локальной сборке.

Проверка `npm run test:integrity` защищает от сценария массовой порчи WordPress-контента. Она
падает, если текст записи стал пустым или слишком коротким, если запись потеряла больше 70%
длины или слов, если исчезло больше 5 файлов, появилось больше 20 новых файлов или изменилось
больше 25 текстовых hash за один запуск. Единичные изменения hash выводятся как warning и не
ломают сборку. В GitHub Actions `Update WordPress content` проверка запускается после импорта
и до `npm run build`; `npm run integrity:update` выполняется только после успешной проверки и
коммитит обновленный baseline вместе с контентом.

Full import с `--prune` может легитимно удалить старые переходные `*-ru.md` для английских
WordPress URL. Integrity-проверка считает такие отсутствующие записи разрешенной чисткой, если
в baseline у записи `language: "ru"` и `sourceUrl` содержит `/en/`; после успешной проверки
новый baseline сохраняется уже без этих переходных файлов.

Текущий отчет “Неполные поля” проверяет только те поля, которые уже реально импортируются:

- фото;
- описание человека;
- теги;
- эпоху;
- дату произнесения для русскоязычных записей.

Дата произнесения извлекается в `statementDate` из русскоязычного нижнего служебного блока
рядом с местом произнесения и источниками. Отсутствие `statementDate` у англоязычных записей
и временных `*-ru.md` с английским `sourceUrl` не должно попадать в отчет “Неполные поля”,
пока для английского потока не описан отдельный workflow. Суд, город, место произнесения,
источник, ссылка на описание дела и автор фото пока не проверяются как ошибки, потому что этих
данных нет отдельными структурными полями. Сборка должна падать только на критических ошибках.
Некритические проблемы лучше выводить отчетом.

Отчет “Языки оригинала” ищет русскоязычные записи, где тело текста содержит явный маркер
оригинального языка: например, “Оригинал последнего слова на кыргызском языке” или
“Последнее слово в оригинале на азербайджанском языке”. Отчет показывает код языка,
человеческое название языка и строку-доказательство. Это промежуточный контрольный инструмент:
он помогает проверить кандидатов перед тем, как автоматическое заполнение `originalLanguage`
будет включено в импорт.

## Связи и переводы

Переводы лучше хранить как отдельные записи, а связывать через стабильный ключ.

Возможные поля:

```yaml
translationGroupId: "12345"
language: "en"
originalLanguage: "ru"
```

Связанные последние слова лучше хранить отдельно от импортируемого текста, если WordPress не
дает надежной структуры.

Вариант:

```text
src/data/relations.json
```

Пример:

```json
[
  {
    "from": 12345,
    "to": 67890,
    "type": "same_case"
  }
]
```

Так импорт из WordPress сможет обновлять тексты, не стирая ручные связи.

## Локальные инструменты

Если нужен интерфейс для ручного связывания слов, его лучше считать dev-инструментом, а не частью production-сайта.

Возможные варианты:

- отдельная Astro-страница, которая доступна только локально;
- отдельный Node.js CLI;
- маленький локальный сервис для записи в `relations.json`.

На GitHub Pages этот инструмент должен быть скрыт или работать только в режиме просмотра.

## Поиск

Поиск должен быть статичным.

Лучший путь:

1. Во время сборки собрать поисковый индекс из Markdown-файлов.
2. Положить индекс в `dist`.
3. В браузере искать по готовому индексу.

Такой подход не требует сервера и хорошо подходит для GitHub Pages.

## Рекомендуемая структура проекта

```text
src/
  components/
  content/
    last-words/
  layouts/
  lib/
  pages/
public/
  images/
    last-words/
scripts/
  fetch-wordpress.mjs
  import-wordpress.mjs
data/
  wp-export/        # ignored, временные JSON-выгрузки WordPress
faq/
  goal.md
  architecture.md
  pages.md
  roadmap.md
```

Контент проекта хранится в `src/content/last-words/` и валидируется через Astro Content
Collections.

## Решения на сейчас

- Astro подходит как основной фреймворк.
- Production должен быть полностью статичным.
- WordPress остается источником истины.
- Первый стабильный формат контента - Markdown + frontmatter.
- `wordpressId` обязателен для каждой записи.
- Ручные связи лучше хранить отдельно от импортируемых файлов.
- SQLite не нужен для первой версии, но может появиться как build-time инструмент.
- Поиск, статистика и проверки должны строиться из локальной копии данных, а не дергать
  WordPress из браузера.

## Что не делать в первой версии

- Не строить отдельную админку.
- Не писать изменения из GitHub Pages.
- Не добавлять серверный backend.
- Не усложнять хранение данных до появления реальных проблем.
- Не делать WordPress API runtime-зависимостью опубликованного сайта.
