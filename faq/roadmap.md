# Дорожная карта

Roadmap описывает порядок развития readonly-дашборда. Цель и продуктовые принципы описаны
в `faq/goal.md`, архитектура и стек - в `faq/architecture.md`.

## Уже сделано

- Astro-проект собирается как статичный сайт под GitHub Pages base path `/final-statement`.
- Добавлен layout, базовая навигация и общий визуальный стиль.
- Подключены Astro Content Collections.
- Контент хранится в `src/content/last-words/` как Markdown + frontmatter.
- Реализован двухшаговый импорт:
  - `fetch:wp` сохраняет JSON из WordPress REST API;
  - `import:wp` генерирует Markdown из локального JSON.
- Импортированы 20 тестовых записей из сохраненного WordPress JSON.
- Сделаны страницы:
  - главная;
  - база слов;
  - страница последнего слова;
  - отчет “Неполные поля”;
  - экспериментальная страница всех имен;
  - страница “О проекте”;
  - 404.
- На странице последнего слова выводятся фото, страна/эпоха, время чтения, теги, описание
  человека, описание дела и HTML-текст WordPress.
- Отчет показывает записи без фото, описания человека, тегов и эпохи.

## Версия 1: надежное зеркало

Главная задача первой версии - получить устойчивое статичное зеркало материалов.

Осталось:

- довести импорт до полного набора опубликованных записей;
- решить, где хранить JSON-выгрузки WordPress: в репозитории, в artifacts или вне git;
- проверить импорт на большем объеме данных;
- добавить защиту от дубликатов `wordpressId`;
- добавить отчет о результате импорта;
- привести страницу “О проекте” к нормальному публичному тексту;
- проверить мобильную верстку основных страниц;
- настроить публикацию на GitHub Pages.

## Версия 2: структура и навигация

Главная задача второй версии - сделать базу удобной для просмотра и первичного анализа.

Должно быть:

- рабочий поиск, фильтры и сортировка на странице `/words/`;
- фильтр по тегам;
- фильтр по стране;
- фильтр по эпохе;
- фильтр по наличию фото;
- страница всех тегов;
- страница конкретного тега;
- страница статистики;
- группировка по странам, эпохам и языкам;
- решение, оставлять ли страницу `/names/` как отдельный способ навигации.

## Версия 3: качество данных

Главная задача третьей версии - расширить отчеты и сделать качество данных наблюдаемым.

Должно быть:

- отчет по дубликатам `wordpressId`;
- отчет по пустым или слишком коротким текстам;
- отчет по тегам с разным написанием;
- отчет по отсутствующим переводам;
- отчет по дефолтным локальным полям, которые требуют ручной настройки;
- экспорт отчетов в JSON или CSV.

## Версия 4: переводы и связи

Главная задача четвертой версии - аккуратно поддержать разные языки и связи между словами.

Должно быть:

- отдельные записи для переводов;
- связь переводов через `translationGroupId`;
- указание языка оригинала;
- переключатель доступных языков на странице последнего слова;
- fallback: если нет перевода на язык сайта, показывается оригинал;
- связанные последние слова: одно дело, апелляция, несколько слов одного человека;
- локальный способ редактировать связи без WordPress.

## Версия 5: поиск

Главная задача пятой версии - быстрый full-text поиск по опубликованному статичному сайту.

Должно быть:

- поисковый индекс, который генерируется во время сборки;
- поиск по именам, делам, тегам и текстам последних слов;
- быстрые результаты в браузере;
- объяснение совпадения: почему найден конкретный текст;
- корректная работа без сервера.

## Версия 6: инструменты анализа

Главная задача шестой версии - дать команде более глубокие рабочие инструменты поверх базы.

Должно быть:

- расширенная статистика по базе;
- проверка орфографии;
- отчеты по подозрительным или битым связям;
- экспорт выбранных слов в нужном формате;
- подборки и тематические страницы;
- отдельные представления для редакторской, исследовательской и публичной работы.

## Возможные будущие направления

- Синхронизация с базой ОВД-Инфо или другими внешними базами.
- SQLite как build-time слой для сложной нормализации и отчетов.
- Локальный инструмент для ручного связывания последних слов.
