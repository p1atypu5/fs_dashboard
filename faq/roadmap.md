# Дорожная карта

Roadmap описывает порядок развития readonly-дашборда. Цель и продуктовые принципы описаны
в `faq/goal.md`, архитектура и стек - в `faq/architecture.md`.

## Уже сделано

- Astro-проект собирается как статичный сайт под GitHub Pages base path `/fs_dashboard`.
- Добавлен layout, базовая навигация и общий визуальный стиль.
- Подключены Astro Content Collections.
- Контент хранится в `src/content/last-words/` как Markdown + frontmatter.
- Реализован двухшаговый импорт:
  - `fetch:wp` сохраняет JSON из WordPress REST API;
  - `import:wp` генерирует Markdown из локального JSON.
- `fetch:wp` и `import:wp` поддерживают режим `--all` для полного набора записей.
- Импортированы 503 канонические записи из сохраненного WordPress JSON: 450 на русском и
  53 на английском.
- Импортер очищает WordPress HTML от блоков “Поделиться в соцсетях”.
- Импортер извлекает `statementDate` из нижнего служебного блока рядом с местом произнесения и
  источниками; поддержаны русские, английские, ISO-like и числовые даты.
- Импортер определяет язык конкретной записи (`language`) по тексту записи, а не по языку
  произнесения последнего слова.
- Добавлена защита от дубликатов `wordpressId`: импортер падает до записи файлов, если дубли
  есть во входном JSON или существующем Markdown-контенте; `npm run build` автоматически
  запускает `validate:content`.
- Добавлен `npm test` / `npm run test:stats`: snapshot-проверка статистики главной страницы
  сверяет общее количество канонических слов, разбивку по языкам, количество русскоязычных
  людей и список стран с `scripts/statistics-expectations.json`.
- Добавлен `npm run test:integrity`: проверка целостности очищенных текстов по
  `scripts/content-integrity-baseline.json`. Workflow обновления WordPress-контента запускает
  эту проверку после импорта и обновляет baseline только после успешной проверки.
- Добавлен инкрементальный JSONL-журнал импорта `src/data/wpImportLog.jsonl`: одна строка на
  каждое `create`, `update` или `delete`; сервисная страница `/service/import-log/` показывает
  журнал с фильтрами.
- Сделаны страницы:
  - главная;
  - база слов;
  - страница последнего слова;
  - отчет “Неполные поля”;
  - отчет по изображениям;
  - справочник уникальных имен;
  - страницы людей;
  - библиотека тегов;
  - страницы тегов;
  - страница “О проекте”;
  - 404.
- На странице `/words/` есть русскоязычная статичная таблица: 450 записей, поиск по видимым
  колонкам, фильтры по стране, эпохе и тегу, сортировка по колонкам, сохранение
  поиска/сортировки/фильтров в браузере и кнопка сброса.
- Главная страница считает каноническую базу как 503 слова: 450 русскоязычных и 53
  англоязычных; дублирующие временные `*-ru.md` с английским `sourceUrl` не учитываются в
  счетчиках русской рабочей выборки.
- На странице последнего слова выводятся фото, технические данные фото, страна/эпоха, время
  чтения, теги, описание человека, описание дела и HTML-текст WordPress.
- На странице `/reports/tags/` выводятся уникальные русскоязычные теги, количество записей и
  поиск по тегам; англоязычные теги скрыты.
- Для каждого русскоязычного тега есть отдельная страница `/reports/tags/[slug]/` со списком
  связанных русскоязычных слов.
- На странице `/reports/names/` выводятся уникальные русскоязычные имена, поиск по именам и
  счетчик видимых имен; для каждого человека есть отдельная страница `/reports/names/[slug]/`
  со всеми его русскоязычными словами.
- Отчет показывает записи без фото, описания человека, тегов, эпохи и даты произнесения; дата
  проверяется только для русскоязычной рабочей выборки.
- Отчет “Языки оригинала” группирует русскоязычные записи с найденным другим языком оригинала,
  показывает причины определения языка и сводку найденных языков.

## Версия 1: надежное зеркало

Главная задача первой версии - получить устойчивое статичное зеркало материалов.

Осталось:

- решить, нужны ли JSON-выгрузки WordPress как GitHub Actions artifacts; в репозитории они не
  хранятся;
- проверить мобильную верстку основных страниц.

## Версия 2: структура и навигация

Главная задача второй версии - сделать базу удобной для просмотра и первичного анализа.

Сделано:

- русскоязычная база слов стала основной страницей по умолчанию;
- англоязычные слова скрыты из основной базы и основных отчетов по именам, тегам и датам;
- в отчетах по именам и тегам по умолчанию показываются только русскоязычные имена и теги;
  английские данные не удаляются, а откладываются до отдельного режима или отдельного
  представления.

Осталось:

- страница статистики;
- группировка по странам, эпохам и языкам;
- решение, нужна ли группировка имен по первой букве.

## Версия 3: переводы и язык оригинала

Главная задача третьей версии - ввести модель переводов и языка оригинала, но работать на этом
этапе только с русскоязычным потоком.

Должно быть:

- определить правила для `originalLanguage` и `translationGroupId` на реальных переводах;
- добавить поля в схему контента и импорт с консервативными дефолтами;
- после появления явных языковых полей заменить временные эвристики фильтрации по `sourceUrl`
  и кириллице на фильтрацию по структурным данным;
- не считать все английские записи переводами автоматически;
- найти русскоязычные записи, у которых потенциально есть перевод, и вывести их в отдельный
  отчет кандидатов;
- вручную подтвердить первые группы переводов через `translationGroupId`;
- на странице русскоязычного слова показать язык записи, язык оригинала и доступные переводы,
  если связь уже настроена;
- добавить проверки качества переводов: нет группы у перевода, больше одного оригинала в
  группе, конфликт языков, одиночная группа без связей.

## Версия 4: качество данных

Главная задача четвертой версии - расширить отчеты и сделать качество данных наблюдаемым.

Должно быть:

- отчет по дубликатам `wordpressId`;
- отчет по пустым или слишком коротким текстам;
- отчет по тегам с разным написанием;
- отчет по отсутствующим переводам;
- отчет по дефолтным локальным полям, которые требуют ручной настройки;
- отчет “Неполные поля” должен проверять отсутствие `statementDate` только у русскоязычных
  записей, потому что текущий парсер даты ориентирован на русскоязычный нижний служебный блок;
- отдельная страница ошибок по разным параметрам данных:
  - английские значения в русскоязычных словах и русские значения в англоязычных словах;
  - некорректно заполненные поля: дата произнесения, время чтения, язык, страна, эпоха;
  - значения, которые выглядят как служебные рубрики или регионы вместо стран;
- экспорт отчетов в JSON или CSV.

## Версия 5: структурные поля из нижнего блока

Главная задача пятой версии - постепенно вынести из WordPress-текста важные служебные данные в
отдельные поля frontmatter.

Должно быть:

- место произнесения последнего слова;
- источник;
- ссылка на описание дела;
- автор фото;
- проверки качества по отсутствующим или подозрительным значениям этих полей;
- вывод новых полей на странице последнего слова и в сервисных отчетах.

## Версия 6: английские записи

Главная задача шестой версии - отдельно продумать рабочий процесс для англоязычных записей,
переводов и связанных отчетов.

Должно быть:

- отдельная база англоязычных слов на странице, похожей на русскую базу;
- отдельные правила для английских имен, тегов и служебных данных;
- решение, показывать ли английские имена и теги отдельными отчетами или режимом в текущих
  отчетах;
- отдельные проверки качества для англоязычных записей;
- связь с русскоязычными оригиналами через модель переводов.

## Версия 7: поиск

Главная задача седьмой версии - быстрый full-text поиск по опубликованному статичному сайту.

Должно быть:

- поисковый индекс, который генерируется во время сборки;
- поиск по именам, делам, тегам и текстам последних слов;
- быстрые результаты в браузере;
- объяснение совпадения: почему найден конкретный текст;
- корректная работа без сервера.

План реализации:

1. Определить поля индекса: `title`, `person`, `caseDescription`, `personDescription`,
   `country`, `period`, `tags`, `statementDate`, `language`, `url` и очищенный текст слова.
2. Добавить build-time endpoint `src/pages/search-index.json.ts`, который читает
   `last-words` и генерирует статичный JSON.
3. Вынести очистку текста от HTML и лишних пробелов в отдельную утилиту.
4. Сделать первый минималистичный поиск без внешней библиотеки: поиск по строкам в готовом
   JSON-индексе.
5. Добавить страницу `/search/`: поле поиска, счетчик результатов, карточки найденных слов и
   быстрый переход к странице слова.
6. Показывать причину совпадения: имя, описание дела, описание человека, тег или текст.
7. Добавить фильтры поверх результатов: язык записи, страна, эпоха, тег, наличие фото.
8. Если индекс станет слишком большим или поиск будет недостаточно удобным, перейти на
   MiniSearch или Fuse.js, не добавляя серверную зависимость.
9. Проверить production-сборку: `search-index.json` должен лежать в `dist`, а поиск должен
   работать как полностью статичная страница на GitHub Pages.

## Версия 8: инструменты анализа

Главная задача восьмой версии - дать команде более глубокие рабочие инструменты поверх базы.

Должно быть:

- расширенная статистика по базе;
- проверка орфографии;
- отчеты по подозрительным или битым связям;
- экспорт выбранных слов в нужном формате;
- подборки и тематические страницы;
- отдельные представления для редакторской, исследовательской и публичной работы;
- ревизия повторяющихся UI-блоков: найти элементы, которые переиспользуются, но еще не
  выделены в отдельные Astro-компоненты.

## Возможные будущие направления

- Синхронизация с базой ОВД-Инфо или другими внешними базами.
- SQLite как build-time слой для сложной нормализации и отчетов.
- Локальный инструмент для ручного связывания последних слов.
